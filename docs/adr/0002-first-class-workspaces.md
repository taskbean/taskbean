# Split logical projects from physical workspaces

Taskbean models a Project as the logical rollup for work, while a Workspace is a physical checkout, worktree, fork, or folder where that work happens. This split prevents generated Copilot worktrees and forked session folders from becoming standalone Projects while still preserving the physical paths needed for attribution, skill installation, cleanup, and diagnostics. Agent session ingestion should resolve or create the logical Project and Workspace from session `cwd` metadata when possible, so generated worktrees do not remain orphaned until manual tracking. Tracking exists at both levels: a Project appears in default rollups when it has at least one tracked Workspace, while each Workspace controls physical-path behavior. Project deletion or untracking is logical metadata cleanup by default and must not remove Workspace files unless the user chooses a separate physical cleanup action. Existing path-based Project rows that are really worktrees or session folders should be merged into the logical Project by preserving their paths as Workspaces before duplicate Project rows are hidden or removed.
